## What we covered

### Beyond Frontend: Full Stack Overview
- **Frontend vs Backend** — frontend (HTML/CSS/JS) runs in the browser and handles what users see; backend (Python, Node.js, etc.) runs on a server and handles business logic, data processing, and authentication
- **APIs revisited** — APIs are the contract between frontend and backend; most use the RESTful style with GET (retrieve), POST (create), PUT (update), and DELETE methods
- **Databases** — why `localStorage` isn't enough for real apps: it's device-only, limited to ~10MB, and can't be shared between users; real apps use centralized databases
- **SQL vs NoSQL** — SQL databases (MySQL, PostgreSQL) use structured tables and are good for relational data like banking; NoSQL databases (MongoDB, Firebase) use flexible JSON-like documents and are better for social media or IoT use cases
- **Backend languages** — Node.js, Python, Java, PHP, Go; Node.js is a natural next step since we already know JavaScript — same language, different environment
- **Authentication** — how login works: frontend sends credentials, backend checks the database, creates a token (JWT), and the client includes that token in future requests to prove identity
- **Frontend frameworks** — why plain JS sometimes isn't enough for large apps; React (Facebook), Vue, Angular (Google), and Svelte each solve state management and UI efficiency differently
- **Full Stack combinations** — common stacks like MERN (MongoDB, Express, React, Node) and JAMstack; Next.js as the modern approach that combines React frontend and backend API routes in a single project with easy Vercel deployment
- **Static vs server rendering** — static sites (what we've built) generate HTML at build time and are fast but not personalized; server-rendered apps (Next.js) generate HTML per request and can respond to the logged-in user

### Deployment and Custom Domains
- **Three parties** — registrar (sold you the domain name), DNS (maps the name to a server IP address), and host (runs your actual code); they're often the same company but don't have to be
- **GitHub Pages** — good for static HTML/CSS/JS projects; not suitable for anything needing a server, hidden API keys, or a database
- **Vercel** — serves both static and dynamic sites, supports environment variables to safely hide API keys from the browser, auto-deploys on every `git push`, free for personal projects
- **Environment variables on Vercel** — store `OPENAI_API_KEY` (or any key) in Settings → Environment Variables; the server reads the key so the browser never sees it, solving the API key exposure problem from client-side code
- **Custom domains** — optional but useful for personal branding; GitHub Student Developer Pack gives a free 1-year `.me` domain through Namecheap
- **Connecting a domain to Vercel** — add the domain in Vercel first (it tells you the exact DNS records needed), then add those A records or CNAME records in Namecheap's Advanced DNS tab, delete any default parking records, and wait for propagation
- **DNS propagation** — changes can take anywhere from 5 minutes to 48 hours; use `whatsmydns.net` or `nslookup` to check; Vercel auto-issues a free SSL certificate once DNS resolves

## Key takeaways

- Everything we've built this semester is the frontend half of a full stack; the backend is a separate layer that handles data, authentication, and business logic — usually built in a different course (OIM3640) or on the job
- Vercel is the right choice for the Final Project if you're calling any API that needs a key hidden from the browser — which means almost everyone should use it over GitHub Pages for the final
- Environment variables are not just a "nice to have" — they're the correct way to handle secrets in any deployed app; hardcoding keys in JS files is a real security risk
- A custom domain is completely optional for Demo Day; a working `vercel.app` URL is a real, shareable URL
- DNS is not instant — plan ahead if you want a custom domain live by Demo Day; don't set it up the night before
- For the pest prediction project, the quality of the Claude output depends heavily on the prompt — giving it structured context (location, temperature, humidity, season) produces a much more useful response than just asking "are there pests?"
- Chaining APIs (weather data → Claude analysis) is a powerful pattern: one API provides real-world data, and the LLM turns that data into a human-readable interpretation

## Reflection
Everything we've built this semester — HTML, CSS, JavaScript, API calls — is the frontend layer, and there's an entire parallel world on the server side that handles the things users never see. Knowing where the boundary is (and that Next.js blurs it) makes it easier to understand why some things aren't possible in a static site, like safely hiding an API key.

The deployment session was immediately practical. The three-party analogy (registrar = city, DNS = post office, host = house) finally made DNS click — you can own an address without anyone knowing where the house is until the address book is updated. The Vercel environment variables setup also solved the problem from Session 22 in a cleaner way: instead of hiding a key in a gitignored file that still lives on the client, the key never leaves the server.

The async work session on the pest prediction project was the most hands-on problem-solving of the week. Getting the weather API and Claude API to work together required thinking carefully about what data to pass and how to frame the prompt. The breakthrough was realizing that Claude gives much better pest risk output when you give it structured environmental context — location, temperature, humidity, recent rainfall — rather than asking vaguely. It's essentially turning an API data pipeline into an AI-powered decision tool, which feels like exactly what the course has been building toward.
